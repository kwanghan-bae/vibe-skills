# 🎭 작업 모드 구분 (Action Modes)

> 급발진 방지: READ(분석)와 WRITE(수정)를 명확히 구분한다.

---

## 1. 모드 분류 (Mode Classification)

모든 요청을 먼저 분류:

| 모드 | 키워드 | 허용 작업 |
|:---|:---|:---|
| 🔍 **READ** | "분석", "확인", "보여줘", "조회" | 읽기만 |
| ✏️ **WRITE** | "만들어", "수정", "고쳐", "구현" | 파일 변경 |
| ❓ **AMBIGUOUS** | "이거 좀", "체크" | 질문으로 명확화 |

---

## 2. 모드 전환 규칙

| 상황 | 대응 |
|:---|:---|
| READ → READ | ✅ 즉시 실행 |
| WRITE → WRITE | ✅ TDD 사이클 진입 |
| READ → WRITE 필요 | 🛑 **STOP** → "수정할까요?" 질문 |
| AMBIGUOUS | 🛑 **STOP** → "어떤 작업을 원하시나요?" 질문 |

### 문제 발견 시 템플릿
```
❗️ [문제명] 발견

📋 내용: [설명]
🔧 제안: [해결 방법]

❓ 수정 작업을 진행하시겠습니까?
```

---

## 3. 급발진 방지 패턴

### ❌ 절대 금지
- **Silent Execution**: READ 요청인데 자동으로 파일 수정
- **Scope Creep**: 요청 범위를 벗어난 추가 작업
- **Assumption Acting**: 명시적 허가 없이 "아마도" 수정

### ✅ 올바른 패턴
- **Explicit Ask**: 불확실하면 질문
- **Minimal Scope**: 요청받은 것만 정확히
- **Report First**: 분석 결과 먼저 보고, 수정은 허가 후

---

## 4. Gemini 전용 강화

> Gemini는 급발진 경향이 특히 강함

### 🚨 필수 브레이크
1. "분석" 키워드 → 코드 생성 절대 금지
2. "확인" 키워드 → 파일 수정 절대 금지
3. 문제 발견 → 보고만, 자동 수정 금지
4. "고쳐줘" 명시될 때까지 대기

### 응답 전 자문
- [ ] 사용자가 정말 이걸 원하는가?
- [ ] 내가 READ를 WRITE로 바꾸지 않았는가?
- [ ] 멈추고 물어봐야 하지 않는가?

---

## ⚠️ 핵심 원칙

1. **"분석해줘" ≠ "고쳐줘"** - 명시적 허가 필수
2. **READ = 읽기 전용** - 수정 절대 금지
3. **의심스러우면 질문** - 추측하지 말고 물어보기

---

## ⚠️ IMPORTANT REMINDER

**급발진은 에이전트의 가장 치명적인 실수다.**

사용자가 "분석해줘"라고 말했는데 코드를 수정하는 순간, 신뢰가 깨진다.
항상 READ와 WRITE를 구분하고, 불확실하면 질문한다.
